## 一、我的核心判断

读完需求和 dr-chen、dr-wang 的讨论后，我有一个强烈的观点要先说清楚：

**"简单 ML"这个标签有误导性风险。**

如果设计不慎，场景 B 会退化成一个调参练习。真正有价值的简单 ML 任务应该测试**科学推理能力**，而不是 hyperparameter 搜索能力。我的设计原则：任务必须能区分"做了实验"和"理解了为什么"。

## 二、场景 B（简单 ML）详细设计方案

### 2.1 核心任务类型

从 PaperWritingBench 中选取含有 ML 实验的论文，设计以下类型的任务：

**类型 B1：方法改进任务**
- 输入：论文提出的方法 M（如带正则化的线性分类器）+ 其使用的数据集规格
- 要求：在保持资源预算（CPU 时间 ≤ 30 分钟，内存 ≤ 8GB）的前提下，提出能够显著改进 M 的方法 M'
- 成功标准：M' 在相同测试集上显著优于 M（需统计检验，p < 0.05），且方法可复现

**类型 B2：理论-实验桥梁任务**（响应 dr-chen 的建议）
- 输入：一篇分析收敛性/泛化界/样本复杂度的理论论文
- 要求：在提供的数据集上验证或反驳该论文的理论预测，并解释差距原因
- 成功标准：实验设计合理，分析深度（不只是复现，而是理解偏差来源）

**类型 B3：消融实验设计任务**
- 输入：一篇报告了几个 trick 的经验性论文（如数据增强、特征工程）
- 要求：设计严谨的消融实验，量化每个 trick 的贡献
- 成功标准：实验设计覆盖所有关键因素，结论与数据一致

### 2.2 数据来源

从 PaperWritingBench 中筛选满足以下条件的论文：
- 包含可下载的公开数据集（UCI、Kaggle、sklearn 内置等）
- 数据规模适合资源受限环境（< 100K 样本）
- 任务类型：分类（二分类/多分类）、回归、聚类、排序
- 方法类型：传统 ML（SVM、随机森林、XGBoost、线性模型等）

**排除标准**：
- 数据集需要特殊许可的
- 依赖 GPU 的基线方法（保证资源受限可执行）
- 2019 年以前发表的已被充分研究的论文

### 2.3 评价指标

| 维度 | 权重 | 评估方式 |
|------|------|----------|
| 性能提升（vs. 论文基线） | 25% | 自动计算，需通过统计显著性检验 |
| 方法合理性 | 20% | 评审：改进有理论/直觉支撑，非随机搜索 |
| 实验严谨性 | 20% | 自动+评审：数据划分、种子固定、超参搜索规范 |
| 可复现性 | 15% | 自动：代码可运行，结果可复现 |
| Ethics 合规 | 10% | 见下节 |
| 结果诚实性 | 10% | 评审：负面结果同样计分，不能只报告最好结果 |

**特别说明**：性能提升权重刻意压低到 25%，就是为了防止"调参赢得高分"的问题。一个解释清楚"为什么方法 M 失败"的 agent 比一个只提交更高准确率的 agent 更有价值。
### 2.4 资源约束规格（可执行性保证）

```
CPU: ≤ 4 cores, 30 分钟
内存: ≤ 8 GB
磁盘: ≤ 5 GB
网络: 禁止（防止调用外部 API 作弊）
Python 包: 白名单制（sklearn, numpy, pandas, scipy, xgboost, lightgbm 等）
```

这些约束直接排除了深度学习方法，强迫 agent 在传统 ML 范围内创新。

## 三、对其他成员的回应

### 对 dr-chen 的建议
完全同意"理论-实践桥梁任务"的价值，已将其设计为类型 B2。这类任务天然要求 agent 理解论文的理论声明，然后设计实验验证——这正是科学推理的核心。

### 对 dr-wang 的 Ethics 框架
我的场景需要特别关注以下 Ethics 点，请 dr-wang 确认：

1. **公平性指标强制要求**：对于涉及人口特征的数据集（如 Adult Census、German Credit），必须报告 demographic parity 和 equalized odds，不能只报告 accuracy。这是我支持的**强制门控**，不是加分项。

2. **数据集偏见披露**：agent 必须在提交报告中说明所用数据集的已知偏见，即使该偏见不影响其方法。

3. **负向结果保护**：benchmark 评分规则必须明确保护负向结果——一个诚实报告"我尝试了 X 但它不起作用，原因是 Y"的 agent 不应得分低于一个只报告成功案例的 agent。

### 对 dr-chen 的验证者问题
对于我的场景，验证框架相对清晰：
- 性能指标：自动计算（无争议）
- 统计显著性：scipy.stats 自动检验
- 代码可运行性：Docker 容器中自动执行
- 方法合理性：需要 LLM 评审 + 人工抽样

## 四、我的关键问题（待团队讨论）

1. **论文-数据集匹配**：PaperWritingBench 的论文是否都有对应可获取的数据集？如果论文使用私有数据，该任务是否可以用类似公开数据集替代？

2. **基线定义**：每个任务的基线是"论文报告的原始方法"还是"我们独立实现的版本"？两者差异可能很大。

3. **评审成本**：方法合理性和实验严谨性维度需要人工评审——这在 benchmark 规模下是否可行？还是依赖 LLM-as-judge？

4. **难度层次**：B1/B2/B3 三类任务的难度不同，是否需要分级评分（简单任务满分 10，困难任务满分 20）？
