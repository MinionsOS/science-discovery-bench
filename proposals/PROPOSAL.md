## 1. Benchmark 整体设计理念

### 1.1 设计目标

MinionsOS2-ScienceBench 旨在测量 AI agent 系统在**真实科研任务**中的能力，重点考察：
- 从已有研究成果出发，提出有价值的改进或扩展
- 在计算资源受限条件下完成有效科学探索
- 符合 AI Ethics 标准的研究行为

### 1.2 与现有 Benchmark 的差异

| 特性 | PaperBench | PaperWritingBench | **MinionsOS2-ScienceBench** |
|------|-----------|-------------------|----------------------------|
| 核心任务 | 复现论文 | 撰写论文 | 改进/扩展已有研究 |
| 评估导向 | 准确性 | 写作质量 | 科学创新性 |
| Ethics 嵌入 | 无 | 无 | **系统内置，强制审核** |
| 结果验证 | 自动化 | 人工 | 分层（自动+人工+形式化） |
| 资源约束 | 无上限 | 无上限 | **明确预算限制** |

### 1.3 MinionsOS2 特色

本 benchmark 的差异化优势在于将 **Ethics 审核** 作为一等公民嵌入评估流程：

1. 每个任务附带 Ethics 检查清单，由 dr-wang 角色的 agent 执行
2. Ethics 审核通过是得分的必要条件（Layer 1 门控）
3. 引入**可解释性维度**：agent 必须说明其方法的推导过程

## 2. 场景 A：理论研究

**主要负责人**: a:dr-chen

### 2.1 任务描述

基于 PaperWritingBench 中具有明确算法/理论贡献的论文，要求 agent 完成以下类型任务之一：

**A1 — 复杂度改进**
- 输入：论文中的 O(f(n)) 算法描述 + 问题形式化定义
- 输出：O(g(n)) 改进算法（g < f）的证明或实现
- 示例：给定 O(n²) 的近似算法，探索 O(n log n) 是否可行

**A2 — 定理推广**
- 输入：论文中的定理 T 及其前提条件 C
- 输出：放宽条件 C' ⊊ C 下定理 T' 的证明，或反例说明无法推广

**A3 — 反例构造**
- 输入：论文中提出但未证明的猜想 Q
- 输出：Q 的反例，或 Q 成立的证明框架

### 2.2 数据来源

筛选 PaperWritingBench 中满足以下条件的论文：
- 领域：近似算法、在线算法、图论算法、组合优化、信息论
- 条件：论文有明确复杂度/正确性声明，结果可用形式化语言重述

**目标数量**: 15-20 个任务实例

### 2.3 评价指标

| 维度 | 权重 | 评估方式 |
|------|------|----------|
| 形式正确性 | 40% | 自动验证（Lean4/程序验证器）或专家二审 |
| 技术创新度 | 30% | 与已知最优解的差距（由领域专家评定） |
| 证明清晰度 | 20% | 结构化评分（逻辑链完整性、步骤标注） |
| Ethics 合规 | 10% | Ethics 审核框架（见第 5 节） |

**阶梯式评分**（Partial Credit）：
- 完全正确的最优证明：100%
- 正确但非最优：60-80%（取决于差距大小）
- 方向正确但有漏洞：20-40%
- 完全错误：0%

### 2.4 Ethics 审核点

- **A-E1**: 归因透明性 — 是否标注推导来源（训练知识 vs. 新推理）
- **A-E2**: 污染标记 — 若源论文在已知训练集中，任务标记为 "potentially contaminated"
- **A-E3**: 引用合规 — 任务描述和结果均需标注原论文来源

## 3. 场景 B：简单机器学习

**主要负责人**: a:prof-liu（本节为 dr-chen 草稿，待 prof-liu 审定修改）

### 3.1 任务描述

基于 PaperWritingBench 中包含传统 ML 方法的论文，要求 agent 在资源受限条件下提出并实现改进的 ML 方案。

**B1 — 方法替换与改进**
- 输入：论文的数据集描述 + 基线 ML 方法（如逻辑回归、决策树）+ 评估指标
- 输出：改进的 ML 方案（特征工程 + 模型选择 + 超参数策略）+ 实验结果
- 约束：仅可使用 scikit-learn 等标准库，禁止深度学习框架

**B2 — 理论-实践验证**
- 输入：论文中的收敛性/泛化性理论声明 + 对应数据集
- 输出：在实际数据上验证或反驳该理论预测的实验设计与结果
- 价值：比纯粹调参更接近科学发现

**B3 — 公平性改进**
- 输入：论文中使用敏感属性（性别/种族/年龄）的 ML 方案
- 输出：在保持性能的前提下降低公平性差距的改进方案

### 3.2 数据来源

筛选条件：
- 使用公开数据集（UCI、Kaggle 标准数据集，避免版权问题）
- 论文中有明确的 baseline 性能数字
- 任务规模：数据集 < 100K 样本，特征 < 500 维

**目标数量**: 15-20 个任务实例

### 3.3 评价指标

| 维度 | 权重 | 评估方式 |
|------|------|----------|
| 性能提升 | 35% | 相对 baseline 的改进幅度（统计显著性 p < 0.05） |
| 实验设计质量 | 25% | 交叉验证、消融实验、对比合理性 |
| 可复现性 | 20% | 固定随机种子，代码可运行 |
| Ethics 合规 | 10% | 见第 5 节 |
| 负向结果价值 | 10% | 证明某方法无效也得分（须有严谨实验支撑） |

### 3.4 Ethics 审核点

- **B-E1**: 数据公平性 — 检查敏感属性的差异化影响
- **B-E2**: P-hacking 防范 — 是否预注册假设或使用多重比较校正
- **B-E3**: 数据来源合规 — 数据集使用符合其许可协议

## 4. 场景 C：深度学习

**主要负责人**: a:alex（本节为 dr-chen 草稿，待 a:alex 审定修改）

### 4.1 任务描述

基于 PaperWritingBench 中的深度学习架构论文，要求 agent 在明确的计算预算约束下提出架构改进。

**C1 — 架构效率改进（CV）**
- 输入：图像分类/检测论文的模型架构描述 + 数据集 + 基线性能
- 约束：推理 FLOPs 不超过基线的 80%（或参数量减少 20%）
- 输出：满足约束的改进架构 + 性能报告

**C2 — 架构效率改进（NLP）**
- 输入：文本分类/序列标注论文的 Transformer 架构 + 基线性能
- 约束：训练时间 ≤ 基线的 120%（或显存占用 ≤ 基线 80%）
- 输出：满足约束的改进方案 + 消融分析

**C3 — 跨域迁移**
- 输入：在领域 X 表现优秀的架构
- 输出：迁移到领域 Y 的适配方案及实验验证

### 4.2 数据来源

筛选条件：
- 使用标准 benchmark 数据集（ImageNet 子集、GLUE、CIFAR）
- 论文代码公开，可复现
- 单实验 GPU 时间 ≤ 4 小时（在 V100/A100 上）

**目标数量**: 10-15 个任务实例（DL 任务成本更高，数量较少）

### 4.3 评价指标

| 维度 | 权重 | 评估方式 |
|------|------|----------|
| 性能（效率约束内） | 35% | 相对 baseline 的指标提升 |
| 架构创新性 | 25% | 专家评审（非暴力搜索的设计洞察） |
| 计算效率 | 20% | FLOPs/参数量/训练时间的实测对比 |
| Ethics 合规 | 10% | 见第 5 节 |
| 复现性 | 10% | 独立运行验证 |

### 4.4 Ethics 审核点

- **C-E1**: 计算碳排放报告 — 需报告估算碳排放（使用 codecarbon 或等效工具）
- **C-E2**: 偏见评估 — CV/NLP 任务需报告在不同人群/语言上的公平性
- **C-E3**: 模型能力边界披露 — 明确报告改进方案的失败模式

## 5. Ethics 审核框架

**主要负责人**: a:dr-wang（本节为 dr-chen 草稿，待 dr-wang 审定，dr-wang 拥有否决权）

### 5.1 三层审核机制

**Layer 1 — 强制门控（未通过则任务得分为 0）**

| 检查项 | 适用场景 | 验证方式 |
|--------|----------|----------|
| 数据来源合法 | A/B/C | 自动检查数据集许可协议 |
| 无有害内容 | A/B/C | 自动 + 人工抽检 |
| 引用来源存在 | A/B/C | 自动验证论文 DOI/arXiv ID |
| 计算资源在预算内 | B/C | 自动监控 |

**Layer 2 — 评分扣减（扣除该场景最终得分的 10-20%）**

| 检查项 | 扣分幅度 | 验证方式 |
|--------|----------|----------|
| 未披露 AI 辅助程度 | -15% | 人工审核输出 |
| 结果无法复现 | -20% | 独立运行验证 |
| 未进行统计显著性检验 | -10% | 自动分析代码/报告 |
| 忽略论文假设条件 | -15% | 专家评审 |

**Layer 3 — 标记警示（记录但不影响得分）**

- 高碳排放（> 某阈值 kg CO2e）
- 数据集存在已知偏见未讨论
- 结论过度泛化

### 5.2 Ethics 审核流程

```
任务提交
  └─> Layer 1 自动检查 ──[失败]──> 得分 = 0，返回违规报告
        └─[通过]
          └─> Layer 2 自动检查 ──> 计算扣分
                └─> Layer 3 标记检查
                      └─> 综合得分 = 场景得分 × (1 - Layer2扣分率)
                            └─> 生成 Ethics 审核报告（附最终提案）
```

### 5.3 MinionsOS2 系统集成

在 MinionsOS2 中，Ethics 审核由专门的 `a:dr-wang` 角色 agent 执行：
- 该 agent 拥有**任务阻断权**：可在任何阶段暂停评分流程
- 所有 Ethics 判决记录在任务链上，可审计
- 阻断意见必须在团队讨论后解决（至少 3/4 同意继续）

## 6. 实施方案

### 6.1 技术可行性分析

| 组件 | 可行性 | 关键依赖 | 风险 |
|------|--------|----------|------|
| 场景 A 形式验证 | 中等 | Lean4 / 人工专家 | 专家资源稀缺 |
| 场景 B ML 实验 | 高 | scikit-learn, Python | 低风险 |
| 场景 C DL 训练 | 中等 | GPU 资源 | 计算成本 |
| Ethics 自动检查 | 高（Layer 1-2） | codecarbon, 规则引擎 | 覆盖率有限 |
| 基准污染检测 | 低-中 | 训练数据元信息 | 各 agent 训练集不透明 |

### 6.2 资源需求评估

**计算资源**:
- 场景 A：无 GPU 需求，主要依赖 LLM 推理
- 场景 B：CPU 集群，单任务 < 1 小时
- 场景 C：每任务 1-4 GPU 小时（V100/A100 等效）
  - 总计约 15 任务 × 4 小时 = 60 GPU 小时

**人力资源**:
- 领域专家评审（场景 A 尤为关键）：约 20 人/天
- 初始任务集构建：约 2 周
- Ethics 审核规则编写：约 1 周

### 6.3 时间规划（从启动算起）

| 阶段 | 时间 | 输出物 |
|------|------|--------|
| 需求冻结 + 任务筛选 | 第 1-2 周 | 任务候选列表 |
| 场景 A 任务集构建 | 第 2-4 周 | 15-20 个理论任务 |
| 场景 B 任务集构建 | 第 2-3 周 | 15-20 个 ML 任务 |
| 场景 C 任务集构建 | 第 3-5 周 | 10-15 个 DL 任务 |
| Ethics 框架实现 | 第 2-4 周 | 自动检查脚本 + 规则库 |
| 小规模试运行 | 第 5-6 周 | Pilot 报告 |
| 修订与最终版本 | 第 7-8 周 | v1.0 发布 |

## 7. 评估框架

### 7.1 总体评分结构

```
综合得分 = Σ(场景权重 × 场景得分 × Ethics合规系数)

场景权重（暂定，待团队确认）:
  - 场景 A（理论）: 35%
  - 场景 B（ML）:   30%
  - 场景 C（DL）:   35%

Ethics合规系数 = max(0, 1 - Layer2扣分率)
  （Layer 1 失败直接使该任务得分 = 0）
```

### 7.2 多维度评价矩阵

| 维度 | 场景 A | 场景 B | 场景 C | 工具/方法 |
|------|--------|--------|--------|-----------|
| 技术正确性 | 形式验证 | 统计检验 | 独立复现 | 自动 |
| 创新性 | 专家评审 | 与 SOTA 对比 | 架构新颖度 | 人工 |
| 可复现性 | Lean4 证明 | 种子固定运行 | Docker 镜像 | 自动 |
| 伦理合规 | 归因检查 | 公平性测试 | 碳排放报告 | 混合 |
| 负向结果 | 反例价值 | 无效实验说明 | 失败模式分析 | 人工 |

### 7.3 Leaderboard 设计

- 主榜：按综合得分排名
- 子榜：按场景分类（适合单专项评估）
- Ethics 榜：专门展示 Ethics 合规得分（激励正向行为）
- 透明度要求：所有提交的 Ethics 审核报告公开可查
