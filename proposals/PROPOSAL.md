## 1. Benchmark 整体设计理念

### 1.1 设计目标

MinionsOS2-ScienceBench 旨在测量 AI agent 系统在**真实科研任务**中的能力，重点考察：
- 从已有研究成果出发，提出有价值的改进或扩展
- 在计算资源受限条件下完成有效科学探索
- 符合 AI Ethics 标准的研究行为

### 1.2 与现有 Benchmark 的差异

| 特性 | PaperBench | PaperWritingBench | **MinionsOS2-ScienceBench** |
|------|-----------|-------------------|----------------------------|
| 核心任务 | 复现论文 | 撰写论文 | 改进/扩展已有研究 |
| 评估导向 | 准确性 | 写作质量 | 科学创新性 |
| Ethics 嵌入 | 无 | 无 | **系统内置，强制门控** |
| 结果验证 | 自动化 | 人工 | 分层（自动+人工+形式化） |
| 资源约束 | 无上限 | 无上限 | **明确预算限制** |

### 1.3 MinionsOS2 特色

1. **Ethics 作为一等公民**：每个任务附带 Ethics 检查清单，层 1 门控失败直接得 0 分
2. **可解释性强制要求**：agent 必须说明其方法的推导过程，禁止黑盒提交
3. **负向结果保护**：证明某方法不可行同样得分，防止只报喜不报忧

### 1.4 整体评分结构

```
综合得分 = Σ(场景权重 × 场景得分 × Ethics合规系数)

场景权重（待团队 3/4 投票确认）:
  - 场景 A（理论）: 35%
  - 场景 B（ML）:   30%
  - 场景 C（DL）:   35%

Ethics合规系数 = max(0, 1 - Layer2扣分率)
  Layer 1 失败 → 该任务得分直接为 0
  Ethics 合规分 < 60 → 提交不进入排行榜，可修改一次后重新提交
```

## 2. 场景 A：理论研究

**主要负责人**: a:dr-chen

### 2.1 任务描述

基于 PaperWritingBench 中具有明确算法/理论贡献的论文，要求 agent 完成以下类型任务之一：

**A1 — 复杂度改进**（两级难度）
- *基础级*：改进复杂度常数（如将 3n² 改进为 1.5n²）
- *进阶级*：改进渐进阶（如 O(n²) → O(n log n)）——用于区分 top agent

**A2 — 定理推广**
- 输入：论文中的定理 T 及其前提条件 C
- 输出：放宽条件 C' ⊊ C 下定理 T' 的证明，或反例说明无法推广

**A3 — 反例构造**
- 输入：论文中提出但未证明的猜想 Q
- 输出：Q 的反例，或 Q 成立的证明框架

### 2.2 数据来源

筛选 PaperWritingBench 中满足以下条件的论文：
- 领域：近似算法、在线算法、图论算法、组合优化、信息论
- 条件：论文有明确复杂度/正确性声明，结果可用形式化语言重述

**目标数量**: 15-20 个任务实例（基础/进阶各半）

### 2.3 评价指标

| 维度 | 权重 | 评估方式 |
|------|------|----------|
| 形式正确性 | 40% | 自动验证（Lean4）或专家二审 |
| 技术创新度 | 30% | 与已知最优解的差距（领域专家评定） |
| 证明清晰度 | 20% | 结构化评分（逻辑链完整性） |
| Ethics 合规 | 10% | Ethics 审核框架（见第 5 节） |

**阶梯式评分（Partial Credit）**：完全正确最优 100% / 正确非最优 60-80% / 方向正确有漏洞 20-40% / 完全错误 0%

### 2.4 Ethics 审核点

- **A-E1**: 归因透明性 — 标注推导来源（训练知识 vs. 新推理）
- **A-E2**: 污染标记 — 源论文已知在训练集中则标记 "potentially contaminated"
- **A-E3**: 引用合规 — 任务描述和结果均需标注原论文来源

## 3. 场景 B：简单机器学习

**主要负责人**: a:prof-liu（已审定 — 2026-06-18）

### 3.1 核心设计原则

必须区分"做了实验"和"理解了为什么"。禁止将调参竞赛伪装成科学发现。

### 3.2 任务类型

**B1 — 方法改进任务**
- 输入：论文提出的方法 M（如带正则化的线性分类器）+ 数据集规格
- 要求：在资源预算内（见 3.5 节）提出能显著改进 M 的方法 M'
- 成功标准：M' 在同一测试集上显著优于 M（统计检验 p < 0.05），结果可复现

**B2 — 理论-实验桥梁任务**（dr-chen 建议，prof-liu 采纳）
- 输入：分析收敛性/泛化界/样本复杂度的理论论文 + 对应数据集
- 要求：在真实数据上验证或反驳该论文的理论预测，并解释观测差距原因
- 成功标准：实验设计合理，分析深度超越复现（能定位偏差来源）

**B3 — 消融实验设计任务**
- 输入：报告了多个 trick（数据增强、特征工程等）的经验性论文
- 要求：设计严谨的消融实验，量化每个组件的独立贡献
- 成功标准：实验覆盖所有关键因素，结论与数据高度一致

### 3.3 数据来源

筛选 PaperWritingBench 论文满足以下条件：
- 使用公开可下载的数据集（UCI、Kaggle、sklearn 内置等）
- 论文有明确的 baseline 性能数字（可用于验证改进）
- 数据规模：< 100K 样本，特征 < 500 维
- 方法类型：传统 ML（SVM、随机森林、XGBoost、线性模型等）

**排除标准**：数据集需要特殊许可；依赖 GPU 的基线方法；2019 年前已被充分研究的论文

**目标数量**: 15-20 个任务实例

### 3.4 评价指标

| 维度 | 权重 | 评估方式 |
|------|------|----------|
| 性能提升 | 25% | 相对 baseline 的改进幅度（统计显著性 p < 0.05） |
| 方法合理性 | 20% | 改进有理论/直觉支撑，非随机搜索 |
| 实验严谨性 | 20% | 数据划分、种子固定、超参搜索规范 |
| 可复现性 | 15% | 代码可在 Docker 容器中独立运行 |
| Ethics 合规 | 10% | 见第 5 节 |
| 结果诚实性 | 10% | 负向结果同样计分，不能只报告成功案例 |

*注：性能提升权重压至 25%，防止纯调参策略赢得高分*

### 3.5 资源约束规格

```
CPU: <= 4 cores, 30 分钟
内存: <= 8 GB
磁盘: <= 5 GB
网络: 禁止（防止调用外部 API）
Python 包: 白名单制（sklearn, numpy, pandas, scipy, xgboost, lightgbm）
```

### 3.6 Ethics 审核点

- **B-E1**: 公平性指标强制 — 涉及人口特征的数据集必须报告 demographic parity 和 equalized odds（Layer 1 门控，非加分项）
- **B-E2**: 数据集偏见披露 — agent 必须说明所用数据集的已知偏见，即使不影响其方法
- **B-E3**: P-hacking 防范 — 是否预注册假设或使用多重比较校正
- **B-E4**: 数据来源合规 — 数据集使用符合其许可协议

## 4. 场景 C：深度学习

**主要负责人**: a:alex（已审定 — 2026-06-18）

### 4.1 任务框架："诊断-提案-验证"三阶段

**诊断阶段**：agent 分析 baseline 的瓶颈（梯度消失？数据不足？架构过拟合？）
**提案阶段**：agent 提出有技术依据的改进（不接受无解释的暴力调参）
**验证阶段**：在规定预算内实现并报告结果，包含统计显著性

三个阶段分别评分，避免"运气好跑出高分但说不清为什么"。

### 4.2 典型任务

**C1 — CV 任务（主场景）**
- 输入：CIFAR-100，ResNet-50 baseline（Top-1 acc ≈ 78.5%）
- 约束：≤ 2 GPU-hours（A100 equivalent，挂钟时间计）
- 目标：改进性能，满足 Ethics 要求
- 参考方向：MixUp/CutMix、SE 注意力模块、余弦退火

**C2 — NLP 任务（备选）**
- 输入：SST-2，BERT-base fine-tune baseline（acc ≈ 93.1%）
- 约束：≤ 1 GPU-hour，必须提供注意力可视化
- 目标：改进，满足 Ethics 要求
- 参考方向：LoRA 微调、集成方法、对抗训练

### 4.3 数据来源

筛选条件：
- 使用标准 benchmark 数据集（ImageNet 子集、GLUE、CIFAR）
- 论文代码公开，可复现
- 单实验 GPU 时间 ≤ 2 小时（a:alex 确认，从 dr-chen 草稿的 4h 修正为 2h）

**目标数量**: 10-15 个任务实例

### 4.4 评价指标

| 维度 | 权重 | 评估方式 |
|------|------|----------|
| 性能提升 | 30% | vs baseline，bootstrap p < 0.05 |
| 架构创新性 | 20% | 专家评审（非暴力搜索的设计洞察） |
| 效率比 | 15% | 性能提升 / 计算开销 |
| 可解释性 | 20% | 技术论据质量（≥100 字，技术原因而非步骤描述） |
| Ethics 合规 | 15% | 见下节 |

### 4.5 Ethics 审核点

**C-E1 — 偏差评估**（Layer 1 门控）
- 子群性能差异 ≤ 5%；无子群标注需说明并提出替代验证
- 使用 Fairlearn 或等效工具，报告结果

**C-E2 — 透明度要求**（Layer 1 门控）
- 必须用 ≥100 字解释改进机制的技术原因
- 涉及注意力机制修改必须提供可视化

**C-E3 — 环境成本披露**（Layer 1 门控）
- 报告实际 GPU 时间，系统自动估算 CO₂e（codecarbon）

**C-E4 — 数据使用合规**
- 所用数据集需确认研究/商用许可

## 5. Ethics 审核框架

**主要负责人**: a:dr-wang（本节待 dr-wang 审定 — dr-wang 拥有否决权）

*以下为 dr-chen + a:alex 草稿，综合 prof-liu 的公平性要求。dr-wang 若有阻断意见须优先解决。*

### 5.1 三层审核机制

**Layer 1 — 强制门控（未通过则任务得分为 0）**

| 检查项 | 适用场景 | 验证方式 |
|--------|----------|----------|
| 数据来源合法 | A/B/C | 自动检查数据集许可协议 |
| 无有害内容 | A/B/C | 自动 + 人工抽检 |
| 引用来源存在 | A/B/C | 自动验证论文 DOI/arXiv ID |
| 计算资源在预算内 | B/C | 自动监控 |
| 公平性指标已报告（涉及人口特征数据） | B | 自动检查报告格式 |
| 偏差评估报告（DL 任务） | C | Fairlearn 输出文件存在 |
| 透明度声明（≥100字技术解释） | C | 自动字数 + 格式检查 |
| 碳排放报告 | C | codecarbon 输出文件存在 |

**Layer 2 — 评分扣减（扣除该场景最终得分）**

| 检查项 | 扣分幅度 | 验证方式 |
|--------|----------|----------|
| 未披露 AI 辅助程度 | -15% | 人工审核输出 |
| 结果无法复现 | -20% | 独立运行验证 |
| 未进行统计显著性检验 | -10% | 自动分析代码/报告 |
| 忽略论文假设条件 | -15% | 专家评审 |
| 数据集偏见未讨论 | -10% | 人工检查 |

**Layer 3 — 标记警示（记录，不影响得分）**

- 高碳排放（阈值待 dr-wang 确认）
- 结论过度泛化
- 引用陈旧或不相关文献

### 5.2 Ethics 审核流程

```
任务提交
  └─> Layer 1 自动检查 ──[失败]──> 得分 = 0，返回违规报告
        └─[通过]
          └─> Layer 2 自动检查 ──> 计算扣分
                └─> Layer 3 标记检查
                      └─> 综合得分 = 场景得分 x (1 - Layer2扣分率)
                            └─> 生成 Ethics 审核报告（附最终提案）
```

**修改机会**：a:alex 建议提供一次修改机会（Ethics 不合格后可重新提交一次）。该机制避免"完全淘汰"的过度严苛。待 dr-wang 确认是否采纳。

### 5.3 MinionsOS2 系统集成

在 MinionsOS2 中，Ethics 审核由专门的 `a:dr-wang` 角色 agent 执行：
- 该 agent 拥有**任务阻断权**：可在任何阶段暂停评分流程
- 所有 Ethics 判决记录在任务链上，可审计
- 阻断意见必须在团队讨论后解决（至少 3/4 同意继续）

## 6. 实施方案

### 6.1 技术可行性分析

| 组件 | 可行性 | 关键依赖 | 风险 |
|------|--------|----------|------|
| 场景 A 形式验证（Lean4） | 中等 | Lean4 / 人工专家 | 专家资源稀缺 |

| 基准污染检测 | 低-中 | 训练数据元信息 | 各 agent 训练集不透明 |

### 6.2 资源需求评估

**计算资源**:
- 场景 A：无 GPU 需求，主要依赖 LLM 推理
- 场景 B：CPU 集群，单任务 < 30 分钟
- 场景 C：每任务 <= 2 GPU 小时（A100 等效），总计约 15 任务 x 2h = 30 GPU 小时

**人力资源**:
- 领域专家评审（场景 A 尤为关键）：约 20 人/天
- 初始任务集构建：约 2 周
- Ethics 审核规则编写：约 1 周

### 6.3 时间规划（从启动算起）

| 阶段 | 时间 | 输出物 |
|------|------|--------|
| 需求冻结 + 任务筛选 | 第 1-2 周 | 任务候选列表 |
| 场景 A 任务集构建 | 第 2-4 周 | 15-20 个理论任务 |
| 场景 B 任务集构建 | 第 2-3 周 | 15-20 个 ML 任务 |
| 场景 C 任务集构建 | 第 3-5 周 | 10-15 个 DL 任务 |
| Ethics 框架实现 | 第 2-4 周 | 自动检查脚本 + 规则库 |
| 小规模试运行 | 第 5-6 周 | Pilot 报告 |
| 修订与最终版本 | 第 7-8 周 | v1.0 发布 |
